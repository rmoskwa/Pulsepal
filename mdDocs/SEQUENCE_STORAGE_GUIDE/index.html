    PulsePal Sequence Storage Guide¶

    Overview¶

    This guide explains how to store Pulseq sequences in the PulsePal database for optimal retrieval and search functionality. The storage system uses a multi-table architecture designed for efficient RAG (Retrieval-Augmented Generation) queries.

    Database Architecture¶

    Table Structure¶

    1. pulseq_sequences_test - Main sequence metadata
    2. sequence_chunks_test - Code segments for granular search
    3. crawled_code_test - Code dependencies (main sequences, helpers, reconstruction)
    4. crawled_docs_test - Documentation and data files

    Key Design Principles¶

    • Flat JSON structure at root level (no nested metadata objects)
    • Intelligent classification by Claude for all semantic fields
    • Programmatic extraction for technical fields (line counts, file sizes)
    • Foreign key relationships ensure data integrity

    Storage Workflow¶

    Step 1: Initial Discovery¶

    # Navigate to repository
cd /path/to/pulseq-explorer/sequences/XXX-sequence-name

# Check for metadata
ls metadata.json notes.md

# Navigate to profiles
cd sequence-profiles/[sequence-name]

# List all dependencies
ls -la

    Step 2: Context Gathering¶

    1. Read README.md for context (DO NOT store)
    2. Identify main sequence file (.m or .py)
    3. List all dependency files (excluding README.md)
    4. Explore cloned repository if needed: 
      /pulseq-explorer/clonedDirectories/[repo-name]

    Step 3: Intelligent Analysis¶

    Sequence Classification¶

    Sequence Family Decision Tree: 

    1. Is it a quantitative mapping protocol?
   → QALAS, MOLLI, MP2RAGE, etc.

2. Is it a clinical architecture?
   → MPRAGE, DIFFUSION, ASL, PRESS, STEAM

3. Is it a novel technique?
   → Create new family (e.g., WAVE-CAIPI, SMS-EPI)

4. Is it a basic readout?
   → FLASH, bSSFP, TSE, EPI, GRE, SE, FID

    Vendor Features Detection: 

    vendor_features = []

# GE-specific
if "TRID" in code:
    vendor_features.append("TRID_segments")
if "pislquant" in code:
    vendor_features.append("pislquant")
if "seq2ceq" or "writeceq" in code:
    vendor_features.append("pge2_conversion")

# Siemens-specific
if "ICE" in code:
    vendor_features.append("ICE_reconstruction")
if "registerGradEvent" in code:
    vendor_features.append("pre_registration")

# Empty array = pure Pulseq

    Critical Rules¶

    File Placement Rules¶

    File Type Table Content Types
    Main sequence crawled_code_test main_sequence only
    Helper functions crawled_code_test helper_function only
    Reconstruction crawled_code_test reconstruction only
    Data files crawled_docs_test data_file
    Config files crawled_docs_test configuration
    Documentation crawled_docs_test Various doc_types

    Path Specifications¶

    # Dependencies list (for RAG search)
"local_dependencies": ["rf90.txt", "trajectory.mat"]  # Filenames only

# Documentation files (for storage)
"file_name": "csv_imports/rf90.txt"  # Relative path from sequence dir

    README.md Handling¶

    • Inside sequence-profiles/: READ for context, DO NOT store
    • Outside sequence-profiles/: Can be stored as documentation

    Common Patterns¶

    6-Section Template (85% of sequences)¶

    1. Parameter Definition
    2. System Configuration
    3. RF/Gradient Creation
    4. Timing Calculations
    5. Sequence Assembly
    6. Validation/Output

    Vendor Detection Examples¶

    % GE-specific
seq.addBlock(mr.makeLabel('SET', 'TRID', 1));  % TRID segments

% Siemens-specific
seq.registerGradEvent(gx);  % Pre-registration

% Pure Pulseq (no vendor features)
seq.addBlock(mr.makeDelay(1e-3));

    Troubleshooting¶

    Issue: Missing Dependencies¶

    Solution: Check cloned repository for additional context: 

    find /pulseq-explorer/clonedDirectories/[repo] -name "*.m" -o -name "*.py"

    Issue: Unknown Sequence Family¶

    Solution: Create new classification if sequence is novel: - Novel k-space sampling → New family - Unique preparation → New family - Hybrid approach → New family

    Issue: Vendor Feature Uncertainty¶

    Solution: Look for these patterns: 

    % Vendor-specific = beyond standard mr.* functions
seq.addBlock(mr.makeLabel('SET', 'TRID', 1));  % GE vendor feature
seq.registerGradEvent(gx);  % Siemens vendor feature
seq.addBlock(mr.makeDelay(1));  % Pure Pulseq (no vendor)

    Issue: File Classification¶

    Decision tree: 

    Is it executable code?
  └─ Yes → Table 3 (crawled_code_test)
      └─ Main sequence? → content_type: "main_sequence"
      └─ Helper? → content_type: "helper_function"
      └─ Reconstruction? → content_type: "reconstruction"
  └─ No → Table 4 (crawled_docs_test)
      └─ Data file? → doc_type: "data_file"
      └─ Config? → doc_type: "configuration"
      └─ Documentation? → doc_type: appropriate type

    Validation Checklist¶

    Pre-Storage¶

    • [ ] All files identified (except README.md in sequence-profiles)
    • [ ] Main sequence file located
    • [ ] Dependencies listed (filenames only)
    • [ ] Vendor features detected
    • [ ] Sequence family determined

    Post-Storage¶

    • [ ] Table 1: Main sequence stored
    • [ ] Table 2: All chunks stored
    • [ ] Table 3: All code dependencies stored
    • [ ] Table 4: All documentation stored
    • [ ] Checklist updated

    Quick Reference¶

    Content Type Matrix¶

    Content Type Table Required Metadata
    Main .m/.py main_sequence 3 sequence_type, complexity, scan_time
    Helper code helper_function 3 purpose, category, functions
    Recon code reconstruction 3 recon_type, input/output format
    CSV/MAT/TXT data_file 4 None (auto-generated)
    Config files configuration 4 None (auto-generated)

    Complexity Levels¶

    1. Level 1: Basic demos, simple concepts
    2. Level 2: Standard clinical sequences
    3. Level 3: Advanced features, acceleration
    4. Level 4: Research sequences, novel methods
    5. Level 5: Highly complex, multiple advanced features

    Storage Command Examples¶

    Basic Storage¶

    # Store a single sequence
python sequence_to_database_storage.py \
  --sequence-path /sequences/001-diffusion/dti-sequence \
  --template-json dti_template.json

    Batch Storage¶

    # Store multiple sequences
for seq in /sequences/*/sequence-profiles/*; do
  python sequence_to_database_storage.py \
    --sequence-path "$seq" \
    --template-json "${seq##*/}_template.json"
done

    Verification Query¶

    -- Check storage completeness
SELECT 
  s.file_name,
  COUNT(DISTINCT c.id) as chunk_count,
  COUNT(DISTINCT d.id) as dep_count,
  COUNT(DISTINCT doc.id) as doc_count
FROM pulseq_sequences_test s
LEFT JOIN sequence_chunks_test c ON s.id = c.sequence_id
LEFT JOIN crawled_code_test d ON s.id = d.coupled_sequences[1]
LEFT JOIN crawled_docs_test doc ON s.id = doc.coupled_sequences[1]
GROUP BY s.id, s.file_name;
